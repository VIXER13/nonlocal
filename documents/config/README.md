# Структура конфигурационных файлов

Для запуска программы NonLocFEM необходимо указать путь к исполняемому файлу и в качестве аргумента передать путь к JSON файлу с конфигурацией запуска, например
```bash
/path/to/NonLocFEM /path/to/config.json
```

Конфигурационный файл `config.json` должен содержать в себе набор определённых ключевых полей, каждое из которых обладает собственной структурой, которая может зависеть от постановки задачи и прочих параметров. Ниже представлены определения основных полей и их структуры.

Примеры конфигурационных файлов представлены в папке [examples](./examples/).

## Основные поля
```json
{
    "task": {...},
    "mesh": {...},
    "save": {...},
    "auxiliary": {...},
    "time": {...},
    "boundaries": {...},
    "materials": {...}
}
```
- [`task`](#task-field) (обязательное) — основная постановка задачи;
- [`mesh`](#mesh-field) (условно обязательное) — параметры сетки: пути к файлам, конфигурации конечных элементов и т.д. (обязательное, когда [`dimension`](#task-field) больше `1`);
- [`save`](#save-field) (опциональное) — параметры сохранения результатов и метаинформации;
- [`auxiliary`](#auxiliary-field) (опциональное) — вспомогательные параметры: правая часть, начальные распределения, доп. условия в некоторых постановках и т.д.;
- [`time`](#time-field) (условно обязательное) — параметры времени в нестационарных расчётах (обязательное, когда [`time_dependency`](#task-field) равно `true`);
- [`boundaries`](#boundaries-field) (обязательное) — граничные условия;
- [`materials`](#materials-field) (обязательное) — параметры материалов.


<a id="task-field"></a>

## Поле `task`
```json
{
    "dimension": <int>,
    "problem": <str>,
    "time_dependency": <bool>
}
```
- `dimension` (обязательное) — размерность задачи, может принимать значения `1` или `2`;
- `problem` (обязательное) — тип задачи, может принимать значения `"thermal"`, `"mechanical"` и `"thermomechanical"`;
- `time_dependency` (обязательное) — зависимость от времени, может принимать значения `true` или `false`.


<a id="mesh-field"></a>

## Поле `mesh`
#### Когда [`dimension`](#task-field) равен `1`
```json
{
    "element_order": <str|int>,
    "quadrature_order": <str|int>
}
```
- `element_order` (опциональное) — порядок элемента, может принимать значения: `"linear"`, `"quadratic"`, `"cubic"`, `"quartic"`, `"quintic"` или числовое значение от `1` до `5` включительно. По умолчанию: `"linear"` (или `1` в числовом представлении).
- `quadrature_order` (опциональное) — порядок квадратуры, может принимать значения: `"linear"`, `"quadratic"`, `"cubic"`, `"quartic"`, `"quintic"` или числовое значение от `1` до `5` включительно. По умолчанию: совпадает с `element_order`.

**Примечание**: В одномерном случае сетка не считывается из файла. Узлы сетки и данные об элементах не хранятся, хранится лишь информация о сегментах из которых состоит рассматриваемое тело. Каждый сегмент выполнен из отдельного материала, поэтому задание параметров сетки и длины сегментов регулируется в поле [materials](#materials-field).

#### Когда [`dimension`](#task-field) равен `2`
```json
{
    "path": <str>
}
```
- `path` (обязательное) — путь к файлу с конечно-элементной сеткой в формате su2.


<a id="save-field"></a>

## Поле `save`
```json
{
    "folder": <str>,
    "config": <str>,
    "csv": <str>,
    "vtk": <str>,
    "precision": <int>
}
```
- `folder` (опциональное) — путь к папке, в которую сохранять результаты. По умолчанию: путь к текущей рабочей директории;
- `config` (опциональное) — имя под которым сохранить текущий конфигурационный файл в папке `folder`. В случае пропуска не сохраняет файл;
- `csv` (опциональное) — имя под которым сохранить результаты расчётов в формате CSV в папке `folder`. В случае пропуска не сохраняет файл;
- `vtk` (опциональное) — имя под которым сохранить результаты расчётов в формате VTK в папке `folder` (когда [`dimension`](#task-field) равен `2`). В случае пропуска не сохраняет файл;
- `precision` (опциональное) — точность чисел с плавающей точкой при сохранении результатов. По умолчанию: использует стандартную точность вывода.

<a id="time-field"></a>

## Поле `time`
```json
{
    "time_step": <float>,
    "initial_time": <float>,
    "steps_count": <int>,
    "save_frequency": <int>
}
```
- `time_step` (обязательное) — размер шага по времени;
- `initial_time` (опциональное) — начальный момент времени. По умолчанию: `0.0`;
- `steps_count` (обязательное) — количество шагов по времени;
- `save_frequency` (опциональное) — частота сохранения результатов. По умолчанию: `1`.


<a id="boundaries-field"></a>

## Поле `boundaries`
#### Когда [`dimension`](#task-field) равен `1`
```json
{
    "left": {...},
    "right": {...}
}
```
- `left` (обязательное) — граничное условие на левой границе;
- `right` (обязательное) — граничное условие на правой границе.
#### Когда [`dimension`](#task-field) равен `2`
```json
{
    "condition_name_1": {...},
    ...
    "condition_name_N": {...}
}
```
- `condition_name_N` (опциональные) — граничные условия. Названия полей совпадают с названиями, определёнными в файле с конечно-элементной сеткой [mesh](#mesh-field). При пропуске поля, граничные условия не применяются на пропущенной границе.

Структура граничных условий, помимо размерности, ещё зависит от типа решаемой задачи.

<a id="thermal-boundaries-field"></a>

### Тепловая задача
```json
{
    "condition_name": {
        "kind": <str>,
        "temperature": <float|str>,
        "flux": <float|str>,
        "heat_transfer": <float>,
        "emissivity": <float>
    }
}
```
- `kind` (обязательное) — определяет род граничного условия, может принимать значения: `"temperature"`, `"flux"`, `"convection"`, `"radiation"`, `"combined"`;
- `temperature` (условно обязательное) — значение температуры или температуры окружающей среды на границе (обязательное, когда `kind` равно `"temperature"` или `"convection"`);
- `flux` (условно обязательное) — значение теплового потока на границе (обязательное, когда `kind` равно `"flux"`);
- `heat_transfer` (условно обязательное) — значение коэффициента конвективного теплообмена (обязательное, когда `kind` равно `"convection"`);
- `emissivity` (условно обязательное) — значение коэффициента излучения (обязательное, когда `kind` равно `"radiation"`)
<details>
<summary>Примеры</summary>

- Температурное граничное условие:
    ```json
    {
        "kind": "temperature",
        "temperature": 487.0
    }
    ```
- Температурное граничное условие может быть задано в виде выражения двух переменных (когда [`dimension`](#task-field) равно `2`)
    ```json
    {
        "kind": "temperature",
        "temperature": "x y: exp(x) + 2 * y"
    }
    ```
- Потоковое условие:
    ```json
    {
        "kind": "flux",
        "flux": -100.0
    }
    ```
- Потоковое граничное условие может быть задано в виде выражения двух переменных (когда [`dimension`](#task-field) равно `2`)
    ```json
    {
        "kind": "temperature",
        "temperature": "x y: exp(x^2 - 1)"
    }
    ```
- Конвективный теплообмен:
    ```json
    {
        "kind": "convection",
        "temperature": 250.0,
        "heat_transfer": 5000.0
    }
    ```
- Температура внешней среды может быть задана в виде выражения двух переменных (когда [`dimension`](#task-field) равно `2`)
    ```json
    {
        "kind": "convection",
        "temperature": "x y: abs(log(x + y))",
        "heat_transfer": 5000.0
    }
    ```
- Температурное излучение:
    ```json
    {
        "kind": "radiation",
        "emissivity": 0.7
    }
    ```
- Комбинированные граничные условия (линейная комбинация теплового потока, конвективного теплообмена и излучения):
    ```json
    {
        "kind": "combined",
        "flux": -100.0,
        "temperature": 250.0,
        "heat_transfer": 5000.0,
        "emissivity": 0.7
    }
    ```
</details>

<a id="mechanical-boundaries-field"></a>

### Механическая задача
```json
{
    "condition_name": {
        "displacement": <float|str>,
        "pressure": <float|str>
    }
}
```
- `displacement` (условно обязательное) — кинематическое граничное условие, исключает использование `pressure` на той же координатной оси на заданной границе;
- `pressure` (условно обязательное) — силовое граничное условие, исключает использование `displacement` на той же координатной оси на заданной границе.
<details>
<summary>Примеры</summary>

Для случая [`dimension`](#task-field) равному `1` на текущий момент нет реализации.

Для случая [`dimension`](#task-field) равному `2`, граничные условия необходимо определять в виде массива условий размерности 2 (по количеству степеней свободы), таким образом, можно определить различные комбинации граничных условий:
- Перемещение границы
    ```json
    {
        "condition_name": [
            { "displacement": 3.0 },
            { "displacement": 2.0 }
        ]
    }
    ```
- Вектор силы, приложенный к границе
    ```json
    {
        "condition_name": [
            { "pressure": 5.0 },
            { "pressure": -4.0 }
        ]
    }
    ```
- Комбинированный вариант
    ```json
    {
        "condition_name": [
            { "displacement": 0.0 },
            { "pressure": -4.0 }
        ]
    }
    ```
- Пропуск условия по одной из осей (равносильно условию `pressure` равно `0`)
    ```json
    {
        "condition_name": [
            null,
            { "displacement": 2.0 },
        ]
    }
    ```
- Перемещение и давление на каждой из осей могут быть определены в виде выражений двух переменных
    ```json
    {
        "condition_name": [
            { "displacement": "x y: abs(y)" },
            { "pressure": "x y: log(x) - sin(y^2)" }
        ]
    }
    ```
</details>

### Термомеханическая задача

В случае термомеханической задачи, для определения температурных и механических граничных условий вместо поля `boundaries` необходимо иметь два поля `thermal_boundaries` и `mechanical_boundaries`, внутренняя структура которых соответствует [тепловой](#thermal-boundaries-field) и [механической](#mechanical-boundaries-field) постановкам соответственно.

<a id="materials-field"></a>

## Поле `materials`

#### Когда [`dimension`](#task-field) равен `1` имеет структуру массива
```json
[
    {
        "elements_count": <int>,
        "length": <float>,
        "physical": {...},
        "model": {...}
    },
    ...
    {
        "elements_count": <int>,
        "length": <float>,
        "physical": {...},
        "model": {...}
    },
]
```
- `elements_count` (обязательное) — количество конечных элементов в материале;
- `length` (обязательное) — длина отрезка с заданным материалом;
- `physical` (обязательное) — физические параметры материала;
- `model` (опциональное) — нелокальные параметры материала. При отсутствии, используется классическая (локальная) модель.

**Примечание**: В одномерном случае левая граница всегда находится в точке 0, сегменты идут последовательно друг за другом, таким образом общая длинна рассматриваемого тела складывается из длин сегментов.

#### Когда [`dimension`](#task-field) равен `2` имеет структуру словаря 
```json
{
    "material_name_1": {
        "physical": {...},
        "model": {...}
    },
    ...
    "material_name_N": {
        "physical": {...},
        "model": {...}
    }
}
```
- `material_name_N` (обязательное) — параметры материала. Названия полей совпадают с названиями, определёнными в файле с конечно-элементной сеткой [mesh](#mesh-field). В случае отсутствия названия у материала его параметры можно определить ключевым полем `DEFAULT`.
- `physical` (обязательное) — физические параметры материала.
- `model` (опциональное) — нелокальные параметры материала. При отсутствии, используется классическая (локальная) модель.

Структура поля `physical` зависит от решаемой задачи, возможные варианты представлены ниже.

<a id="thermal-materials-field"></a>

### Тепловая задача
```json
{
    "conductivity": <float|str|array[float|str]>,
    "capacity": <float>,
    "density": <float>,
    "relaxation_time": <float>
}
```
- `conductivity` (обязательное) — теплопроводность;
- `capacity` (опциональное) — теплоёмкость. По умолчанию: `1.0`;
- `density` (опциональное) — плотность. По умолчанию: `1.0`;
- `relaxation_time` (опциональное) — время релаксации теплового потока. По умолчанию: `0.0`.

<details>
<summary>Примеры</summary>

- Cтационарная задача, изотропный материал
    ```json
    {
        "conductivity": 5.0
    }
    ```
- Нестационарная задача, ортотропный материал
    ```json
    {
        "conductivity": [3.0, 5.0],
        "capacity": 27.0,
        "density": 500.0,
        "relaxation_time": 1.3
    }
    ```
- Стационарная задача, анизотропный материал (полный тензор теплопроволности)
    ```json
    {
        "conductivity": [3.0, 1.0, 
                         1.0, 5.0]
    }
    ```
- Когда [`dimension`](#task-field) равно `1`, коэффициент теплопроводности можно выразить через выражение одной переменной, т.е. зависимость от пространственной координаты
    ```json
    {
        "conductivity": "x: x + 1"
    }
    ```
- Когда [`dimension`](#task-field) равно `1`, коэффициент теплопроводности можно выразить через выражение двух переменных, т.е. зависимость от пространственной координаты и температуры (нелинейная задача)
    ```json
    {
        "conductivity": "x T: abs(T)^3"
    }
    ```
- Когда [`dimension`](#task-field) равно `2`, Компоненты тензора теплопроводности могут быть определены через выражение двух (пространственные координаты) или трёх (пространственные координаты и температура, нелинейная задача) переменных
    ```json
    {
        "conductivity": [
            "x y: (2 * sqr(x) + 20 * sqr(y)) / (sqr(x) + sqr(y))",
            1.0,
            "x y T: abs(T)^3",
            28.0
        ]
    }
    ```

</details>

<a id="mechanical-materials-field"></a>

### Механическая задача
```json
{
    "youngs_modulus": <float|array[float]>,
    "poissons_ratio": <float|array[float]>,
    "shear_modulus": <float>,
    "thermal_expansion": <float|array[float]>
}
```
- `youngs_modulus` (обязательное) — модуль Юнга;
- `poissons_ratio` (обязательное) — коэффициент Пуассона;
- `shear_modulus` (условно обязательное) — модуль сдвига (обязательное, когда материал ортротропный);
- `thermal_expansion` (опциональное) — коэффициент линейного температурного раширения. По умолчанию: 0.0;

<details>
<summary>Примеры</summary>

Механическая задача реализована только в 2D, поэтому все примеры будут рассмотрены только, когда [`dimension`](#task-field) равно `2`.
- Изотропный материал
    ```json
    {
        "youngs_modulus": 210.0,
        "poissons_ratio": 0.3,
        "thermal_expansion": 13e-6
    }
    ```
- Ортотропный материал
    ```json
    {
        "youngs_modulus": [210.0, 252.0],
        "poissons_ratio": [null, 0.3],
        "shear_modulus": 70.0,
        "thermal_expansion": [13e-6, 13.5e-6]
    }
    ```
- Ортотропный материал, альтернативный эквивалетный вариант определения
    ```json
    {
        "youngs_modulus": [210.0, null],
        "poissons_ratio": [0.25, 0.3],
        "shear_modulus": 70.0,
        "thermal_expansion": [13e-6, 13.5e-6]
    }
    ```

**Примечание**: В ортотропном случае необходимо задать только три параметра из 4 среди элементов `youngs_modulus` и `poissons_ratio`, недостающий четвёртый будет вычислен автоматически

</details>

### Поле `model`
```json
{
    "local_weight": <float>,
    "nonlocal_radius": <float|array[float]>,
    "search_radius": <float|array[float]>,
    "distance": <str>,
    "n": <int|float>,
    "influence": <str>,
    "p": <int|float>,
    "q": <int|float>
}
```
- `local_weight` (обязательное) — локальный вес;
- `nonlocal_radius` (обязательное) — радиус нелокального влияния;
- `search_radius` (опциональное) — радиус поиска ближайших соседей. По умолчанию совпадает с `nonlocal_radius`;
- `distance` (опциональное) — функция расстояния, может принимать значения: `"lp"` и `'ellipse_with_rotation'`. По умолчанию: `"lp"`;
- `n` (опциональное) — параметр нормы. По умолчанию: `2`;
- `influence` (опциональное) — функция нелокального влияния, может принимать значения: `"constant"`, `"polynomial"` и `"exponential"`. По умолчанию: `"polynomial"`;
- `p` (опциональное) — параметр функции нелокального влияния. По умолчанию: `1` при [`dimension`](#task-field) равном `1` и `2` при [`dimension`](#task-field) равном `2`.
- `q` (опциональное) — параметр функции нелокального влияния. По умолчанию: `1`.


### Термомеханическая задача

В случае термомеханической задачи, физические параметры материлов необходимо записать в единую структуру `physical`. Модельные параметры могут быть разделены для [тепловой](#thermal-materials-field) и [механической](#mechanical-materials-field) задач, путём определения двух полей `thermal_model` и `mechanical_model`, но так же можно использовать поле `model` для определения одинаковых модельных параметров для обеих задач.
